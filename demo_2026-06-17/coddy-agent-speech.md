## Слайд 3 — Проблема `[01:30–03:10]`

Вот четыре боли, которые я хотел закрыть. Все они перед вами.

**Первая:** облако — это чужой компьютер. Когда вы используете Claude Code или Cursor, ваш код улетает во внешние API. Для enterprise это часто неприемлемо: compliance, NDA, внутренние данные.

**Вторая:** тяжёлые рантаймы. Claude Code требует Node.js, большинство других агентов написаны на Python. В scratch или distroless Docker-образ их не запустишь. А рой агентов в минимальных контейнерах уже давно просится наружу.

**Третья:** нет единого backend. Cursor, ClaudeCode, Codex, OpenClaw, CI-пайплайн, Telegram-бот — у каждого отдельная история, нет общего состояния, каждый клиент делает своё.

**Четвёртая:** on-premise модели. llama.cpp, vLLM, Ollama, DeepSeek, Qwen локально — универсального coding agent harness'а для них просто не существовало.

Coddy закрывает всё это одним решением. [→]

## Слайд 4 — Решение `[03:10–04:20]`

Coddy Agent — это **один статический Go-бинарник**.

35 мегабайт. Никаких зависимостей от рантайма. Скачал — запустил.

Шесть ключевых свойств:

**Distroless-ready** — работает в scratch и distroless образах с read-only rootfs. Идеально для container fleet'ов, где оркестратор управляет ограничениями.

**On-premise first** — данные не покидают ваш контур. Docker Compose — две строки. Или просто запускаете бинарник.

**Multi-provider** — OpenAI, Anthropic, Ollama, любой OpenAI-совместимый API. Меняется одна строка в конфиге, vendor lock-in исключён.

**ACP harness** — это архитектурный принцип. ACP, Agent Control Protocol, — стандарт взаимодействия IDE с AI-агентами. Coddy реализует его, и к нему прикручиваются VS Code, Obsidian, Zed и другие клиенты.

**OpenAI API** — Coddy можно использовать как OpenAI-совместимый backend. Любое внешнее приложение, которое умеет ходить в OpenAI API, может ходить в Coddy.

**Telegram gateway** — встроенный бот с per-user сессиями, ACL и режимами изоляции для групп. Discord и Slack подключаются через тот же adapter interface. [→]

## Слайд 5 — Архитектура `[04:20–05:50]`

Архитектурно это выглядит так.

Снаружи — **клиенты**: Obsidian, Zed, браузер, Telegram-бот, скрипты и CI.

Они подключаются через разные **транспорты**: ACP по stdio, HTTP REST API, или gateway для мессенджеров.

Всё это упирается в **Session Manager** — единое хранилище состояния. Там живут история, rules, skills, режим работы и persistence. Одна сессия может быть одновременно открыта в IDE и в браузере. История синхронизирована.

В центре системы — **ReAct loop**. Это стандартный паттерн для AI-агентов: LLM думает — вызывает инструменты — смотрит на результаты — думает снова. И так до завершения задачи.

Снизу — LLM провайдер, инструменты, MCP серверы, Memory Copilot и cron-like scheduler.

Харнес — это не монолит. Это протокольный каркас, к которому подключается всё остальное. [→]

## Слайд 6 — Возможности `[05:50–06:50]`

Что конкретно умеет Coddy.

**Инструменты**: тридцать с лишним. Чтение, запись, редактирование файлов. Запуск shell-команд. SSH на удалённые хосты — через чистый Go, без внешних зависимостей. Веб-поиск через Google и DuckDuckGo, плюс readability для парсинга страниц.

**Skills** — slash-команды. Пишешь SKILL.md, кладёшь в .coddy/skills — и в интерфейсе и IDE появляется slash-команда. Переиспользуемые рабочие процессы без кода.

**Rules** — агент автоматически подхватывает правила из .cursor/rules, .claude/rules, .codex/rules и .coddy/rules. Написал один раз — работает с любым клиентом без правок.

Дополнительно — long-term memory, глобальная и project-local, с recall и persist пассами вне основного LLM. Cron scheduler запускает задачи по расписанию, хранит задания в markdown и управляется прямо из веб-UI. MCP подключает внешние серверы, от GitHub и файловой системы до баз данных. [→]

## Слайд 10 — Три режима работы `[07:55–08:55]`

И вот важный момент — Coddy умеет работать в трёх режимах.

**Первый: coddy acp** — JSON-RPC 2.0 сервер по stdio. Obsidian или Zed подключают его как external agent. Полный ACP протокол: streaming, permission prompts, восстановление сессий.

**Второй: coddy http** — и здесь самое интересное. Это OpenAI-compatible API. Эндпоинт `/v1/responses`. Это значит — **любой клиент, который умеет работать с OpenAI**, может использовать Coddy как бэкенд. Ваш Python-скрипт, n8n, Open WebUI — всё подключается без изменений. Плюс встроенная React SPA.

**Третий: coddy gateway** — Telegram-бот с per-user сессиями, ACL и isolation modes. Архитектура расширяемая: Discord, Slack — через тот же Gateway adapter interface.

И главное: все три режима используют одно хранилище сессий. IDE + браузер + Telegram синхронизированы автоматически. [→]

## Слайд 12 — Уроки `[09:35–10:35]`

Пять выводов из этого пути — то, что я бы хотел знать на старте.

**Первое — protocol-first побеждает.** Когда ACP заложен с дня 1, HTTP-сервер и Telegram-gateway прицепляются почти бесплатно — поверх того же Session Manager. Интерфейсный контракт переживает реализации. Когда я начинал, я думал, что протокол — это формальность, а основная работа в логике. Оказалось наоборот: всё, что не зафиксировано в контракте, потом приходится переписывать.

**Второе — Go single binary бьёт Node и Python для container-first.** Scratch и distroless работают сразу. В CI всё разворачивается одной командой — никакого «сначала npm install», никаких слоёв с node_modules. Для роя агентов в минимальных образах это даже не оптимизация, это вопрос «можно ли вообще запустить».

**Третье — build tags вместо системы плагинов.** Из одной кодовой базы рождается и lean ACP-бинарник, и полный образ с UI, scheduler, memory и gateway. Это компромисс между монолитом и плагинной архитектурой — нет второй системы сборки, нет несовместимостей версий, всё статически прокомпилировано вместе. Цена — длиннее матрица CI; но она того стоит.

**Четвёртое — совместимость с чужой экосистемой даёт бесплатных пользователей.** Чтение rules из `.cursor`, `.claude`, `.codex` — и человек не переучивается. OpenAI-compatible API — и любой существующий клиент работает без правок. Не выдумывай новые форматы там, где уже есть устоявшиеся: ты не сэкономишь на этом времени, ты только потеряешь пользователей.

**Пятое — shared state требует дисциплины абстракций.** Когда у тебя IDE, браузер и Telegram пишут в одну сессию, любая утечка деталей транспорта в общий слой ломает остальные клиенты. Session Manager пришлось делать самой консервативной частью кодовой базы — каждое его изменение разглядывается через лупу. Если бы делал заново, я бы заложил эту дисциплину сразу, а не пришёл к ней через серию граблей. [→]
