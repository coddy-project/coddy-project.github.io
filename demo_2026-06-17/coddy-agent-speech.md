## Слайд 3 — Проблема `[01:30–03:10]`

Вот четыре боли, которые я хотел закрыть. Все они перед вами.

**Первая:** облако — это чужой компьютер. Когда вы используете Claude Code или Cursor, ваш код улетает во внешние API. Для enterprise это часто неприемлемо: compliance, NDA, внутренние данные.

**Вторая:** тяжёлые рантаймы. Claude Code требует Node.js, большинство других агентов написаны на Python. В scratch или distroless Docker-образ их не запустишь. А container fleet'ы с минимальными образами — это уже не экзотика.

**Третья:** нет единого backend. Cursor живёт в своём мире, CI-скрипты в своём, Telegram-бот в своём. Сессии не шарятся, история не синхронизируется, каждый клиент делает своё.

**Четвёртая:** локальные модели. Ollama, DeepSeek на своём железе — нормального coding agent harness'а для них просто не существовало.

Coddy закрывает всё это одним решением. [→]

## Слайд 4 — Решение `[03:10–04:20]`

Coddy Agent — это **один статический Go-бинарник**.

35 мегабайт. Никаких зависимостей от рантайма. Распаковал — запустил.

Четыре ключевых свойства:

**Distroless-ready** — работает в scratch и distroless/static образах с read-only rootfs. Идеально для container fleet'ов, где оркестратор управляет ограничениями.

**ACP harness** — это архитектурный принцип. ACP, Agent Control Protocol, — стандарт взаимодействия IDE с AI-агентами. Coddy реализует его, и к нему прикручивается всё остальное.

**Multi-provider** — меняется только одна строка в конфиге. Vendor lock-in исключён.

**On-premise first** — данные не покидают ваш контур. Docker Compose — две строки. [→]

## Слайд 5 — Архитектура `[04:20–05:50]`

Архитектурно это выглядит так.

Снаружи — **клиенты**: Cursor, Zed, браузер, Telegram-бот, скрипты.

Они подключаются через разные **транспорты**: ACP по stdio, HTTP REST API, или gateway для мессенджеров.

Всё это упирается в **Session Manager** — единое хранилище состояния. Одна сессия может быть одновременно открыта в IDE и в браузере. История синхронизирована.

В центре системы — **ReAct loop**. Это стандартный паттерн для AI-агентов: LLM думает — вызывает инструменты — смотрит на результаты — думает снова. И так до завершения задачи.

Снизу — LLM провайдер, инструменты, MCP серверы, и опциональные модули: память и планировщик.

Харнес — это не монолит. Это протокольный каркас, к которому подключается всё остальное. [→]

## Слайд 6 — Возможности `[05:50–06:50]`

Что конкретно умеет Coddy.

**Инструменты**: тридцать с лишним. Чтение, запись, редактирование файлов. Запуск shell-команд. SSH на удалённые хосты — через чистый Go, без внешних зависимостей. Веб-поиск и парсинг страниц.

**Skills** — slash-команды. Пишешь SKILL.md, кладёшь в .coddy/skills — и в интерфейсе и IDE появляется slash-команда. Переиспользуемые рабочие процессы без кода.

**Rules** — агент автоматически подхватывает правила из .cursor/rules, .claude/rules, .coddy/rules. Написал один раз — работает с любым клиентом без правок.

Дополнительно — long-term memory с recall и persist пассами, cron-планировщик для плановых запусков, и MCP для подключения внешних серверов. [→]

## Слайд 8 — Три режима работы `[07:30–08:30]`

И вот важный момент — Coddy умеет работать в трёх режимах.

**Первый: coddy acp** — JSON-RPC 2.0 сервер по stdio. Cursor и Zed подключают его как external agent. Полный ACP протокол: streaming, permission prompts, восстановление сессий.

**Второй: coddy http** — и здесь самое интересное. Это OpenAI-compatible API. Эндпоинт `/v1/responses`. Это значит — **любой клиент, который умеет работать с OpenAI**, может использовать Coddy как бэкенд. Ваш Python-скрипт, n8n, Open WebUI — всё подключается без изменений. Плюс встроенная React SPA.

**Третий: coddy gateway** — Telegram-бот с per-user сессиями и ACL. Архитектура расширяемая: Discord, Slack — через тот же Gateway adapter interface.

И главное: все три режима используют одно хранилище сессий. IDE + браузер + Telegram синхронизированы автоматически. [→]

## Слайд 9 — Результаты и планы `[08:30–09:20]`

Подведём итог.

**Что получилось:** 35 мегабайт в scratch-образе. Любой OpenAI-compatible провайдер — GPT-4o, DeepSeek, Qwen локально. Cursor, браузер и Telegram на одной истории сессий. Rules из .cursor/rules работают без изменений.

**Планы:** Discord и Slack адаптеры, vector search в памяти, PWA, готовые образы для container fleet деплоя. [→]
